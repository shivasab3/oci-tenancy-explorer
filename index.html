About The OCI Tenancy Explorer

Tenancy-Wide Visibility And Operational Signals

What This App Does

A lightweight OCI operational review portal for one tenancy at a time.

OCI Tenancy Explorer helps a teammate sync the latest tenancy snapshots, review the estate from multiple angles, drill into a specific OCID, and move from high-level summary to actionable next steps without leaving the app.

What It Surfaces

Tenancy summary, fleet posture, shape inventory, opportunity signals, console announcements, and OCID-level drill-down details.

How It Stays Safe

OCI credentials stay on the host running the collectors. The browser reads generated JSON snapshots and never handles OCI credentials directly.

Start Here

  1. Click Sync Data to refresh all snapshots in sequence.
  2. Use Tenancy Overview to understand overall scope and estate shape.
  3. Use Fleet View to inspect individual assets, maintenance state, and change signals.
  4. Use Shape Explorer, Opportunities, and Console Announcements to guide deeper discussion.
  5. Click any OCID to open the Asset Inspector for direct resource drill-down and OCI Console handoff.

01 Tabs At A Glance

Tenancy Overview

Use this to get an overview of your tenancy.

Shows the tenancy-wide summary, subscribed regions, compartment-access context, and high-level metrics that help you orient before reviewing individual resources.

Best For

A quick executive readout, scope confirmation, and deciding where deeper review should start.

Fleet View

Use this to review individual assets.

Shows the main resource grid with filters, search, reboot and maintenance context, change signals, and the direct jump into the Asset Inspector for an OCID.

Best For

Finding specific hosts, filtering by region or compartment, exporting filtered results, and discussing operational follow-up.

Shape Explorer

Use this to review compute shape inventory.

Surfaces deployed shapes, shape families, regional distribution, and state patterns so the team can quickly see what compute footprint exists today.

Best For

Capacity conversations, modernization planning, identifying older families, and preparing platform migration or optimization discussion.

Opportunities

Use this to review where the tenancy may deserve deeper attention.

Highlights maturity signals, usage concentration, capability checks, and service patterns that can turn tenancy data into operational or architectural discussion points.

Best For

Finding modernization, resilience, observability, security, or platform adoption opportunities to discuss with the customer.

Console Announcements

Use this to review active notices before or during a customer meeting.

Pulls OCI Console Announcements into a local snapshot so banner messages, operational notices, and actionable service communications are easier to review in one place.

Best For

Weekly customer reviews, operational check-ins, and confirming whether current announcements require follow-up.

02 Asset Inspector

The Asset Inspector is the OCID-centric drill-down experience in the app. Click any OCID shown in the UI or search by exact OCID from Fleet View to open richer resource details.

  • Follow related OCIDs from one resource to another without leaving the panel.
  • Copy the OCID or a short resource summary for tickets, notes, or chat.
  • Open OCI Console with a guided handoff so the copied command is ready for Cloud Shell.

03 Sync Data

Use Sync Data when you want a fresh full-app review. It runs the collectors sequentially and updates the snapshots that power each tab.

Order

fleet_data.json, fleet_data_shapes.json, fleet_data_opportunities.json, then fleet_data_announcements.json.

Why Sequential

Cleaner logs, simpler troubleshooting, and more conservative OCI API load during a full refresh.

The header shows the last successful or partial sync time so teammates can see how fresh the current UI is before they begin review.

04 How To Use This App

Step 1

Run Sync Data and confirm the latest sync time in the header.

Step 2

Review Tenancy Overview for scope, then move into Fleet View for concrete assets.

Step 3

Use Shape Explorer and Opportunities to surface modernization and platform discussion points.

Step 4

Close with Console Announcements and any OCID drill-downs that need operational follow-up.

Under The Hood

Collector Model

The portal uses separate Python collectors for fleet, shapes, opportunities, and announcements. The browser renders their generated JSON snapshots.

OCI Data Sources

The app relies on direct OCI APIs including Identity, Compute, networking-related lookups for OCID drill-down, OS Management Hub where available, and other service-specific reads needed by each snapshot.

Snapshot Host Pattern

portal_server.py and the generated snapshot JSON files should live on the same host so the built-in refresh and reload workflow stays simple and deterministic.

Authentication Options

Option 1: Local ~/.oci/config

Use a local OCI config and API key when running the portal from a workstation or laptop.

Local OCI Config Setup

Use ~/.oci/config With An API Key

Use this path when you are running the portal from a laptop, admin workstation, or another local host.

1

Create The OCI Folder

Run mkdir -p ~/.oci and chmod 700 ~/.oci on the same host that will run portal_server.py.

2

Generate Or Place The API Key

Generate an OCI signing key pair locally, or place the approved private key file on disk, for example ~/.oci/oci_api_key.pem.

3

Upload The Public Key

Upload the matching public key in the OCI Console under your user profile and copy the fingerprint OCI returns.

4

Create ~/.oci/config

Populate user, tenancy, fingerprint, region, and key_file, then run chmod 600 ~/.oci/config.

Instance Principal Setup

Run On OCI Compute Without A Local API Key

Use this path when the portal and collector run on an OCI compute instance and you want the host itself to authenticate to OCI.

1

Run On OCI Compute

Deploy the portal on an OCI compute instance. This same host should also run build_fleet_data.py and serve fleet_data.json.

2

Create A Dynamic Group

Place the compute instance in a dynamic group so OCI can recognize it as an authorized principal.

3

Grant Read Policies

Create IAM policies that let the dynamic group inspect compartments and read the OCI services this app queries.

4

Start With Instance Principal Auth

Launch the app using python3 portal_server.py --auth instance_principal. The refresh modal will then run the collector with the same auth method.
